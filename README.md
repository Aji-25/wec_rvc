## Overview

**rvc** is a fully decentralized, peer-to-peer version control system written in Rust. Unlike Git, there is no remote origin or central hosting service. Peers discover each other on the LAN via **mDNS** and exchange commits over an encrypted, multiplexed **libp2p** connection (Noise + Yamux over TCP).

Key properties:

| Property | Detail |
|---|---|
| **Storage** | Content-addressed object store (SHA-256), gzip-compressed JSON blobs |
| **Transport** | libp2p TCP + Noise encryption + Yamux multiplexing |
| **Discovery** | mDNS (zero-config LAN peer discovery) |
| **Merge** | 3-way merge with LCA (Lowest Common Ancestor) detection |
| **Identity** | Persistent Ed25519 keypair per machine (generated by `ring`, stored in `.rvc/config`) |

## How It Works

### Object Store

All data is stored content-addressed under `.rvc/objects/`, mirroring Git's layout:

```
.rvc/objects/
  ab/
    cdef1234...    ← gzip(JSON(Blob | Tree | Commit))
```

Every object is serialized to JSON with `serde_json`, compressed with gzip (`flate2`), and written to a path derived from the first 2 hex characters of its SHA-256 hash. On read, the hash is re-verified to detect corruption.

**Three object types:**

| Type | Fields |
|---|---|
| `Blob` | `content: Vec<u8>` — raw file bytes |
| `Tree` | `entries: Vec<{name, hash}>` — directory snapshot |
| `Commit` | `tree_hash`, `parent_hashes` (0/1/2), `message`, `timestamp`, `author` |

### Peer Discovery (mDNS)

Both the `rvc discover` command and the `rvcd` daemon use **libp2p mDNS** to broadcast and receive peer advertisements on the local network. No configuration is needed — peers appear automatically.

Once a peer is discovered, a **Handshake** request/response is exchanged carrying the peer's human name and its current `HEAD` hash. Discovered peers are cached in `.rvc/peers` (pipe-delimited flat file) with a timestamp so stale entries can be flagged.

### Sync Protocol

Sync is a **3-round request/response** protocol over libp2p's `request-response` behaviour using a custom length-prefixed JSON codec (`RvcCodec`):

```
Client (rvc sync)                 Server (rvcd)
─────────────────                 ────────────────
GetHead          ───────────────► 
                 ◄─────────────── Head(hash)

GetMissingFrom   ───────────────►
(my HEAD hash)   ◄─────────────── MissingHashes([...])

GetObjects       ───────────────►
([hashes])       ◄─────────────── Objects([(hash, gzipped_bytes)])
```

All transferred objects are validated by re-hashing the decompressed bytes before writing. After objects are saved locally, the merge engine determines the outcome.

### Merge Algorithm

`rvc-core::merge_commits` implements a **3-way merge** (similar to Git):

1. Find the **Lowest Common Ancestor (LCA)** via BFS over both commit DAGs.
2. Load the `base`, `local`, and `remote` trees as `HashMap<path, blob_hash>`.
3. For every path in the union of all three trees:
   - **local == remote** → take either (no change)
   - **local == base** → take remote (remote changed it)
   - **remote == base** → take local (local changed it)
   - **all differ** → **conflict**: write `<<<<<<< LOCAL / ======= / >>>>>>> REMOTE` markers
4. Outcomes: `FastForward`, `AlreadyUpToDate`, `MergeSuccess` (auto-merge commit), or `Conflicts` (user must resolve and re-commit).

## Getting Started

### Prerequisites

- [Rust toolchain](https://rustup.rs/) (stable, 2021 edition)
- Two machines on the same LAN (or two processes on the same machine)

### Build

```bash
# Clone the repo
git clone https://github.com/Aji-25/wec_rvc.git
cd wec_rvc

# Build both binaries
cargo build

# Or build optimized release binaries
cargo build --release
```

Binaries are placed in `target/debug/` (or `target/release/`):
- `rvc` — the main user CLI
- `rvcd` — the background peer daemon

## CLI Reference — `rvc`

```
rvc <COMMAND>
```

| Command | Description |
|---|---|
| `init` | Initialise a new `.rvc/` repository in the current directory |
| `commit -m <msg>` | Snapshot the working directory and create a new commit |
| `log` | Walk and print the full commit history from `HEAD` |
| `status` | Show `HEAD`, commit count, active merge, and known peers |
| `discover` | Scan the LAN for peers (~5 s) and refresh the local peer cache |
| `peers` | Print all cached peers |
| `sync <multiaddr>` | Fetch and merge commits from a specific peer address |
| `sync-peer <name\|alias\|id>` | Sync with a cached peer by name, alias, or peer-id prefix |
| `whoami` | Print the configured author name and peer identity |
| `set-author <name>` | Update the author name stored in `.rvc/config` |
| `set-peer-name <name>` | Update the peer name advertised to other devices |
| `alias-peer <peer> <alias>` | Set a local nickname for a cached peer |
| `conflicts` | List files involved in a pending merge conflict |
| `resolve [<files>...]` | Mark conflicted files as resolved (checks no markers remain) |
| `abort-merge` | Abort a merge in progress and restore the local `HEAD` tree |

### Examples

```bash
# Machine A
cd /path/to/project
rvc init
echo "Hello" > README.md
rvc commit -m "Initial commit"
rvcd &                          # start the background daemon
rvc whoami                      # shows your peer name

# Machine B (same LAN)
cd /path/to/project
rvc init
rvc discover                    # finds Machine A
rvc peers                       # lists cached peers
rvc sync-peer peer-a            # fetch and merge from Machine A
rvc log                         # see the merged history
```

## Daemon Reference — `rvcd`

The `rvcd` daemon runs in the background and handles **inbound sync requests** from other peers. Without it, a peer can still push (sync) out to others but cannot receive incoming syncs automatically.

```bash
# Start daemon in the background (requires an initialised repo)
rvcd &

# One-shot LAN scan (prints discovered peer names, then exits)
rvcd discover
```

**What the daemon does:**

- Listens on a random TCP port (`/ip4/0.0.0.0/tcp/0`)
- Broadcasts itself via mDNS and responds to incoming handshakes
- Serves `GetHead`, `GetMissingFrom`, and `GetObjects` requests
- Persists peer cache and its own Ed25519 keypair to `.rvc/config`

## Repository Layout (`.rvc/`)

```
.rvc/
├── config              # key=value pairs: author, peer_name, keypair (base64)
├── objects/            # content-addressed gzipped JSON objects
│   └── ab/
│       └── cdef1234…
├── refs/
│   ├── HEAD            # current commit hash (plain text)
│   ├── MERGE_HEAD      # remote HEAD during an in-progress merge
│   ├── MERGE_TREE      # merged tree hash during a conflict
│   └── MERGE_CONFLICTS # newline-separated list of conflicted paths
├── peers               # pipe-delimited peer cache: name|peer_id|addr|head|timestamp
└── peer_aliases        # peer_id=alias mappings
```

## Code Complexity Assessment

This codebase sits at **intermediate-to-advanced complexity**. Here's a breakdown by file:

### `rvc-core/src/lib.rs` — ★★★★☆ (Advanced)
The most algorithmically rich file. Implements:
- A generic content-addressed object store (SHA-256 + gzip + JSON + hash verification)
- BFS-based DAG traversal to find ancestors and compute LCA
- A full 3-way file-level merge algorithm producing conflict markers
- Correct handling of adds, deletes, and edits across all three sides

This is non-trivial systems code with careful correctness requirements.

### `src/main.rs` — ★★★☆☆ (Intermediate-Advanced)
The largest file (1 380 lines). Covers:
- A complete CLI with 15 subcommands via `clap`
- Asynchronous libp2p swarm management with `tokio::select!` event loops
- A multi-round async sync protocol (GetHead → GetMissingFrom → GetObjects)
- The full merge action pipeline (materialize tree, handle conflict state)
- Peer cache management with fuzzy/alias/prefix resolution
- Merge state persistence (`MERGE_HEAD`, `MERGE_TREE`, `MERGE_CONFLICTS`)

Managing async state machines while coordinating filesystem I/O and network I/O is inherently complex.

### `rvcd/src/main.rs` — ★★★☆☆ (Intermediate)
The background daemon. Handles:
- A persistent Ed25519 identity (keygen via `ring`, stored as base64)
- An event-loop that processes mDNS discovery events and all three sync request types
- Daemon mode (infinite loop) and one-shot discover mode sharing the same swarm builder

### `rvc-core/src/protocol.rs` — ★★☆☆☆ (Intermediate)
A clean, focused implementation of the libp2p `request_response::Codec` trait:
- Length-prefixed (4-byte big-endian) JSON framing
- Separate read/write size limits (16 MiB for requests, 64 MiB for responses)
- Generic over message types via `serde`

### `rvc-core/src/sync.rs` — ★★☆☆☆ (Intermediate)
Wire-protocol types and object-transfer helpers:
- `find_missing`: walks the local commit chain, collects commit + tree + blob hashes not on the remote
- `read_raw_object` / `save_raw_object`: transfers compressed bytes without re-compressing, with hash verification on receive

### `tests/e2e.rs` — ★★★☆☆ (Intermediate)
Full end-to-end integration tests that:
- Spawn real `rvc` and `rvcd` processes using `std::process::Command`
- Parse daemon stdout to extract live listen addresses
- Exercise all merge scenarios: fast-forward, clean 3-way merge, conflict resolution, and abort-merge

| File | Lines | Complexity | Key Concepts |
|---|---|---|---|
| `src/main.rs` | 1 380 | ★★★☆☆ | async libp2p, CLI, merge pipeline |
| `rvc-core/src/lib.rs` | 530 | ★★★★☆ | DAG, 3-way merge, object store |

**Overall verdict:** The code is **genuinely non-trivial**. It correctly implements distributed systems primitives (DAG merge, LCA, content-addressed storage, encrypted P2P transport) without relying on any existing VCS library. A developer comfortable with Rust and async Tokio will find it readable and well-structured; it is not beginner-level code.

cargo test

**Test coverage includes:**
